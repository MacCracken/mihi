# mihi — security audit (2026-05-19)

**Cuts**: pre-0.6.0 (audited surface = 0.5.0 code + ai-hwaccel 2.2.6 transitive).
**Auditor**: mihi maintainer + Claude pair-review session.
**Outcome**: 0.6.0 lands with three internal fixes (C-1, M-1, C-2) + CVE
awareness documentation. No critical vulnerabilities; all findings are
defense-in-depth or transitive-only.

## Scope

| Layer | Surface |
|---|---|
| Direct syscalls | `uname(2)` (via agnosys), `open(2)` + `read(2)` + `close(2)` of `/proc/cpuinfo`, `/proc/meminfo`, `/proc/uptime`, `/etc/os-release`, `/sys/devices/system/cpu/online` |
| Direct parsers | `mihi_parse_cpu_range`, `mihi_parse_cpu_model`, `mihi_find_meminfo_field`, `mihi_parse_meminfo_kb`, `mihi_extract_meminfo_bytes`, `mihi_parse_uptime_secs`, `mihi_find_osrelease_key`, `mihi_parse_osrelease_value` |
| Transitive (ai-hwaccel 2.2.6 under the no-exec mask) | `/sys/class/drm/cardN/device/{vendor,device,product_name,mem_info_vram_total,mem_info_vram_used,driver,hwmon/*,gpu_busy_percent}`, `/sys/class/accel/*` (TPU), `/sys/class/misc/{intel_npu,samsung_npu,mtk_apu,qaic}/`, `/dev/{groq,neuron*}`, PCIe + NUMA sysfs trees, `readlink(2)` |

## Methodology

Two passes, per the
[`feedback-security-audit-web-research`](../../memory) memory: a clean
audit is *not* possible from internal review alone.

1. **Internal review** — per-file walk of `src/*.cyr`, checking:
   - Bounds on every `load8` / `store8` (len-checked, no underflow)
   - Syscall returns (`sys_open < 0`, `sys_read <= 0`)
   - Integer overflow on `v = v * 10 + …` digit accumulators
   - Buffer mutations (NUL writes within `p < len`)
   - TOCTOU: not applicable — every probe is a single open/read/close
2. **External CVE research** — searched CVE/NVD/MITRE for known issues
   against every kernel surface mihi (or ai-hwaccel transitively)
   touches. Targeted 2025 + 2026 because that's the lifetime of the
   kernels mihi consumers will run on.

## Findings — internal

### C-1 — `mihi_parse_cpu_range` accepts descending ranges (low)

**File**: `src/cpu.cyr:31`
**Path**: `mihi_parse_cpu_range`
**Class**: Input validation / data integrity
**Description**: When the input contains a descending range like
`"10-5"`, the parser computes `total + (hi - lo + 1) = total + (-4)`
and the returned CPU count is corrupted (potentially negative if the
descending range comes first). Linux kernel does not emit this shape
on `/sys/devices/system/cpu/online`, but the parser doesn't validate.
**Exposure**: Theoretical — would require an attacker to write to
`/sys/devices/system/cpu/online` (root-only).
**Fix landed in 0.6.0**: `if (hi < lo) { hi = lo; }` — coerce to
single-CPU range rather than reject (matches the spirit of "bare N
treated as N-N").

### M-1 — Unbounded digit accumulation overflow (low)

**File**: `src/mem.cyr:46` (`mihi_parse_meminfo_kb`)
**Class**: Integer overflow on adversarial input
**Description**: `v = v * 10 + (load8(buf + p) - 48)` runs while
digits remain, without any cap on digit count. An attacker who can
write to `/proc/meminfo` could supply a value with 20+ digits and
overflow the i64 accumulator. Realistic only with root (already
compromises the box), but adding the cap is a one-line defensive
measure.
**Exposure**: Theoretical.
**Fix landed in 0.6.0**: Cap digit count at 19 (max i64 is 19
digits). Same fix applied to `mihi_parse_uptime_secs` and the
lo/hi accumulators in `mihi_parse_cpu_range`.

### C-2 — Same digit-overflow class as M-1 (low)

**Files**: `src/cpu.cyr:31` (`mihi_parse_cpu_range` lo/hi),
`src/host.cyr:45` (`mihi_parse_uptime_secs`)
**Class**: Integer overflow on adversarial input
**Description**: Identical shape to M-1 — unbounded `v = v * 10 + …`
loops on data from `/sys` and `/proc`. Same threat model, same fix.
**Fix landed in 0.6.0**: Folded into the M-1 fix.

### G-1 — Trust in ai-hwaccel for NUL-terminated device names (informational)

**File**: `src/gpu.cyr:97` (`mihi_gpu_name`)
**Class**: Trust boundary / dependency contract
**Description**: `mihi_gpu_name(idx)` returns the cstring from
`profile_device_name(p)` directly to the caller. If ai-hwaccel ever
ships a code path that constructs a `device_name` without a
trailing NUL, the consumer's `println()` walks OOB.
**Exposure**: None on 2.2.6 — every `profile_set_device_name`
call site uses either a string literal (NUL-terminated by the
compiler) or the `str_builder_build` / `alloc + memcpy + NUL`
pattern (verified in the 2.2.6 fixes).
**Fix**: None needed. Documented here as a trust-boundary note —
any future ai-hwaccel release must preserve this invariant.

## Findings — external (CVE)

### CVE-2025-40289 — VRAM sysfs attributes crash on GPUs without VRAM (medium, transitive)

**Surface**: `/sys/class/drm/cardN/device/mem_info_vram_total`,
`mem_info_vram_used`
**Source**: ai-hwaccel `src/detect/rocm.cyr` reads both files via
`sysfs_read_int` during `registry_detect_no_exec()`.
**Description**: On AMD GPUs that don't have dedicated VRAM, the
amdgpu driver exposes the sysfs attribute but the kernel crashes
when it's read. The upstream fix hides the attribute on these
GPUs.
**Exposure on archaemenid**: Confirmed working — Ryzen 5800H iGPU
returns `mem_info_vram_total = 3221225472` (3 GiB shared) without
crash. archaemenid kernel `7.0.5-arch1-1` either has the fix or
is on a path the CVE doesn't trigger (some APU configurations
expose VRAM successfully; the CVE is narrower than "all APUs").
**Mitigation**: None on mihi side. ai-hwaccel's `detect_rocm`
already handles a failed `sysfs_read_int` (`if (mem_total <= 0)
{ mem_total = 8 * BYTES_PER_GIB; }`), so a kernel that hides the
file post-fix degrades gracefully. mihi cannot avoid a pre-fix
kernel that crashes when reading the file — that's a kernel bug.
**Recommendation in CHANGELOG**: mihi consumers running on AMD
GPUs should ensure their kernel has CVE-2025-40289 fixes (mainline
Linux 6.15+; check distro for backports). Listed in the 0.6.0
"Known Environmental Issues" section.
**References**:
- [NVD CVE-2025-40289](https://nvd.nist.gov/vuln/detail/CVE-2025-40289)
- [Linux kernel commit a67a9f99ce1306898d7129a199d42876bc06a0f0](https://git.kernel.org/) (one of three fix commits)

### CVE-2025-40288 — `ttm_resource_manager_usage()` NULL deref on APUs (medium, transitive)

**Surface**: `AMDGPU_INFO_VRAM_USAGE` ioctl + sysfs paths that fan
into `ttm_resource_manager_usage()`.
**Source**: ai-hwaccel may transitively touch this path via the
hwmon enrichment in `detect_rocm` (`/sys/class/drm/cardN/device/
gpu_busy_percent`, `gpu_metrics`).
**Description**: `man->bdev` (backing device pointer) remains NULL
on APU platforms where VRAM manager isn't fully initialized.
Subsequent dereference crashes the kernel.
**Exposure on archaemenid**: Not observed during smoke runs.
Same caveat as CVE-2025-40289 — APU configurations that don't
initialize the VRAM manager are a subset of all APUs.
**Mitigation**: Same as CVE-2025-40289 — kernel-side fix; mihi
has no avoidance path. Listed alongside in the Known
Environmental Issues section.
**References**:
- [NVD CVE-2025-40288](https://nvd.nist.gov/vuln/detail/CVE-2025-40288)
- [SUSE security tracker](https://www.suse.com/security/cve/CVE-2025-40288.html)

### Other 2025 amdgpu CVEs (low/no exposure)

Reviewed but not actionable for mihi:
- **CVE-2025-68286** (NULL pointer in DRM display callbacks) — display path; mihi reads no display-state sysfs.
- **CVE-2025-37766** (Azure Linux amdgpu DoS) — Azure-specific code path; not in mainline kernel sysfs surface.
- **CVE-2025-37769** (AMD SMU11 division by zero) — SMU sensor path; mihi reads `mem_info_*` and `gpu_busy_percent` but not SMU-direct paths.
- **CVE-2025-21941** (AMD DRM NULL pointer) — DRM ioctl path; mihi uses sysfs reads, not ioctls.
- **CVE-2025-40354** (AMD display stack bounds) — display stack; not reached.
- **CVE-2025-38362** (`get_first_active_display()` NULL check) — display state; not reached.

### Non-amdgpu surfaces

- **`/proc/cpuinfo`**, **`/proc/meminfo`**, **`/proc/uptime`** — no
  parser-related CVEs found in 2025–2026 affecting *consumer*
  parsing. The kernel-side write paths are stable. Most 2025 Linux
  kernel CVEs (CVE-2025-38352 race in posix-cpu-timers,
  CVE-2025-37979 buffer overflow, etc.) are unrelated to /proc
  string content.
- **`uname(2)`** — no relevant CVEs. The syscall is one of the
  oldest in Linux; the ABI hasn't moved meaningfully since 2.6.
- **`/etc/os-release`** — no parser CVEs found. The freedesktop
  spec is stable and mihi's quote-stripping matches the spec's
  shell-style key=value contract.
- **`readlink(2)` on `/sys/class/drm/cardN/device/driver`** — used
  by ai-hwaccel to confirm the `amdgpu` driver. Standard symlink
  read; the buffer size (256 bytes in ai-hwaccel's `read_symlink`)
  exceeds the max sysfs symlink target. No exposure.

## Deferred / out-of-scope

Nothing deferred to 0.6.x patches from this audit — the in-scope
fixes are small enough that landing them all in 0.6.0 is simpler
than splitting. If post-cut review surfaces something heavier, the
0.6.x patch slots are available.

## Methodology note for future audits

External research saved this audit from rubber-stamping. CVE-2025-40288
and CVE-2025-40289 are both kernel-side bugs in code mihi calls into
transitively — internal review of mihi's source could not have found
them, and internal review of ai-hwaccel's source wouldn't either
(both repos are doing the right thing on their side; the bug is in
the kernel's amdgpu driver). The
[`feedback-security-audit-web-research`](../../memory) memory note
should be honored on every future audit.
